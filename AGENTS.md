## Write Yourself — Never Implement These

The following must always be written by the developer, not generated by you.
This applies in both DISCUSS and IMPLEMENT mode.

If asked to implement any of the following, refuse and instead:
* Explain the concept clearly
* Describe the exact problem it solves
* Point to what the developer needs to understand before writing it
* Offer to review their implementation once written

**Never implement. Never write a partial version to fill in.**

* Idempotency check logic (`INSERT ... ON CONFLICT` pattern, deduplication logic)
* Retry and exponential backoff logic
* Circuit breaker state transitions (closed, open, half-open, probe logic)
* `SELECT FOR UPDATE SKIP LOCKED` usage
* Redis sliding window rate limiting logic
* HMAC signature computation and verification
* Tenant isolation query filters
* Celery task recovery logic (`acks_late`, stuck delivery recovery job)
* RAG pipeline and embedding logic

## Architecture

Strict layered architecture:

API → Service → Data Layer → Database

### Rules

#### API (Routes)

* Handle HTTP concerns only (validation, auth, response formatting)
* No business logic
* No direct DB access

#### Services

* Contain all business logic
* Responsible for:
  * idempotency
  * retries
  * delivery orchestration
  * rate limiting decisions
* Manage transactions
* Call data layer and Celery tasks

#### Data Layer (Unified Queries Layer)

All database access lives inside a single `queries/` layer.

This layer uses a **hybrid approach**:

##### ORM Usage (Control Plane)

* Uses SQLAlchemy Session
* Used for:
  * tenants
  * endpoints
  * configuration
* No concurrency-critical logic

##### Raw SQL Usage (Data Plane)

* Uses asyncpg
* Used for:
  * event ingestion
  * delivery processing
  * idempotency
  * retries
  * locking (`FOR UPDATE SKIP LOCKED`)
* Must be explicit and performance-aware

### Important Rules

* ORM and raw SQL can coexist in the same module, but:
  * must NOT be mixed inside the same function or transaction flow
* Keep functions clearly separated by responsibility:
  * CRUD-style → ORM
  * concurrency-critical → raw SQL

#### Celery Tasks

* Thin wrappers only
* Must call services
* No business logic inside tasks

## Database

* ORM: SQLAlchemy (Session-based)
* Driver for hot paths: asyncpg

### Rules

* ORM is allowed only for control-plane entities
* Raw SQL is required for data-plane and concurrency-critical logic
* Do not mix ORM and raw SQL in the same logical operation
* Every query must:
  * include `tenant_id` filtering where applicable
  * consider index usage
  * be written with scale in mind

### Tenant Isolation Enforcement

Every query you generate that touches tenant-owned data must include a `tenant_id` filter.
If you generate a query without one, flag it immediately:

```python
# FLAG — TENANT ISOLATION: This query is missing a tenant_id filter.
# Every query on tenant-owned data must be scoped to the authenticated tenant.
# Add: .where(Model.tenant_id == current_tenant.id)
```

Never generate a cross-tenant query without explicitly flagging it and explaining why it is intentional.

### Index Awareness

After generating any query that will run on a table expected to grow large, add:

```python
# Run EXPLAIN ANALYZE on this query after seeding test data to confirm index scan.
```

## Transactions

* Transactions are managed at the Service layer

### Rules

* Services:
  * control SQLAlchemy Session lifecycle
  * manage asyncpg connections when needed
* Data layer:
  * must NOT manage transactions

## Data Handling

* ORM returns mapped models
* Raw SQL returns Pydantic models
* Do not return raw DB rows
* Do not leak DB schema outside data layer

## Async Rules

* Entire codebase must be async
* No blocking I/O allowed
* Do not use sync libraries (e.g. requests)
* Use async-compatible libraries only

## Error Handling

* Services raise domain-specific exceptions
* Services must not return HTTP responses

### Central Exception Handling

* All exceptions handled in one place in FastAPI
* All error responses must follow:

```json
{"error": "message", "code": "ERROR_CODE"}
```

## Logging

* Use structured JSON logging via `structlog`

### Log in:

* Services (business events)
* Task execution boundaries
* External I/O (HTTP calls)

### Do NOT log in:

* Data layer
* Routes (except minimal request logging)

### Required Fields on Every Log Line

`tenant_id`, relevant entity IDs (event_id, delivery_id, endpoint_id where applicable).
Timestamp is added automatically by structlog.

```python
logger.info("delivery_attempted", tenant_id=str(tenant_id),
            delivery_id=str(delivery_id), endpoint_url=endpoint.url,
            attempt_number=delivery.attempt_number)
```

## Testing Strategy

### Suggest Tests Alongside Every Code Generation

Every time you generate a non-trivial piece of code, append a test block.
Do not skip this even if the developer did not ask for tests.

Format:
```
--- Tests to write ---
1. Happy path: [what to assert]
2. Failure path: [failure scenario and what to assert]
3. Tenant isolation: [cross-tenant test if code touches tenant-owned data]
```

### Must be implemented immediately for:

* Idempotency logic
* Delivery logic
* Retry behavior
* Tenant isolation — test that tenant A cannot access tenant B's data

### Other tests:

* Can be added after feature completion

## Celery

* Use separate queues:
  * `default` — for fan-out tasks
  * `delivery` — for delivery execution tasks

### Rules

* Tasks must:
  * be idempotent
  * call services
  * not contain business logic
* Use `acks_late=True` on all delivery and critical tasks
* Always flag a Celery task missing `acks_late=True` that performs critical work:

```python
# FLAG — RELIABILITY: This task performs critical work without acks_late=True.
# A worker crash after task pickup but before completion will silently lose this task.
```

## Flags — Generate But Warn

Generate the code but add a visible flag comment when you see:

* A query without an index on a table expected to grow large
* A synchronous operation inside an async function
* A missing timeout on an external HTTP call
* A missing `tenant_id` filter on a tenant-owned query
* A Celery task doing critical work without `acks_late=True`
* A Redis operation that is not atomic where atomicity matters
* A read-then-write pattern without atomic DB guarantees

Format:
```python
# FLAG: [one line description of the issue and why it matters]
```

## Anti-Patterns — Forbidden

* SQL outside the queries layer
* Business logic inside the data layer
* Business logic inside Celery tasks
* Blocking I/O in async code
* Read-then-write without atomic DB guarantees
* Missing `tenant_id` filters in multi-tenant queries
* Implementing Write Yourself concepts (see top of this file)
